## 二面发现（杨瑾龙）

二面深入考察了三个方向：RAG 项目实战、长程 Claude Code / context 压缩、现场 15 分钟 OpenClaw 多租户设计。

### 1. 理论丰富，但从未闭过环

RAG、context 压缩、prompt 优化三个方向呈现出完全一致的模式：

**能说出来的：** 召回率、精准度、幻觉率、语义切片、overlap 参数（PDF 15-25%、聊天 5-10%）、加权裁剪（核心词 3 分、无关词 0 分）、余弦相似度 BERT 检测、元数据镜像、prompt 引擎模块化、CoT、约束前置……

**说不出来的：**
- 这些参数是怎么定的？效果提升了多少？— 没有数据
- RAG 上线后遇到过什么坑？— 一个都说不出
- 上线后相关性下降怎么排查修复？— 答不上来
- Prompt 优化做过 AB 测试吗？— "未做过量化评估，方法来自论文"
- 评测集谁建的？— "算法团队的，应用端了解不多"

候选人在商汤的角色是"按 spec 实现功能的工程执行者"。他参与了 RAG 系统、context 压缩、prompt 工程的搭建，但从未主动追踪过上线后效果、没有排查过质量问题、没有建立过反馈循环。理由是"ToG 项目完成合同功能即可，算法优化由算法团队负责"。

**三个方向，同一个问题：只做到了"实现"，从未走到"验证 → 发现问题 → 迭代改进"。**

### 2. Claude Code 操作熟练，但工程判断力不足

候选人确实会用 Claude Code — 配了插件（superpowers 等）、懂 worktree 并发编辑、能让 Claude Code 跑 4-5 小时。操作层面比多数候选人熟练。

但现场 15 分钟 OpenClaw 多租户设计暴露了更根本的问题：

- **没有需求澄清** — 拿到题目直接开干，没问隔离粒度、支持哪些 channel、已有架构等关键问题
- **没有源码调研** — 没有读 OpenClaw 的代码或文档，不了解它实际支持什么
- **把 Claude Code 当搜索引擎** — 让 AI 上网搜，OpenClaw 是小众工具网上没有准确信息，产出了一份和实际架构无关的幻觉方案
- **没有验证产出** — 没有用源码交叉验证 AI 生成的内容是否正确
- **15 分钟没给出有效方案** — 从一开始方向就跑偏了

这里缺失的不是工具操作能力，而是使用 AI 工具的工程最佳实践：
1. **先调研再动手** — 对陌生工具，必须先喂源码和文档给 AI，不能让它上网猜
2. **先澄清再执行** — spec coding 的第一步是理解需求和约束
3. **给 AI 设边界** — 越陌生的领域越要用本地 context 框住 AI，而不是让它自由发挥
4. **验证 AI 产出** — 和 ground truth（源码/文档）交叉验证

本质：会开车，但不会看地图。能让 Claude Code 跑很久，但如果方向一开始就错了，跑再久也是浪费。

### 3. 诚实坦率

两轮面试多次主动承认不足，不包装不吹嘘。这一点是真实的加分项，在创业团队中意味着协作时不会隐藏问题。
